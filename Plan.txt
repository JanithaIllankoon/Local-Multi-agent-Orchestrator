# Local Multi-Agent Orchestrator Implementation Plan

## 1. Project Goal

Build a local multi-agent orchestration system where the user interacts with only one interface: the **Orchestrator**.

The Orchestrator should internally manage multiple local LLM/VLM agents, delegate subtasks to them sequentially, aggregate their outputs, decide what actions to take, and return a single polished response to the user.

The system must be designed so that future hardware upgrades require only changing the model configuration, not rewriting the architecture.

The system should support:

* Local model backends through llama.cpp or compatible OpenAI-style HTTP endpoints.
* A supervisor model for planning and final decisions.
* Two coding models for implementation and review.
* A reasoning critic model.
* An optional contrarian or low-refusal critic model.
* A vision model for screenshots and GUI understanding.
* A deterministic tool executor for filesystem, shell, screenshot, and GUI control.
* A web-based UI for interacting only with the Orchestrator.
* Agent trace visibility for debugging.
* Safety gates outside the model layer.
* Model swapping through config files.

The user should never need to directly talk to specialist models.

---

## 2. Core Design Principle

The architecture must separate **thinking**, **suggesting**, and **acting**.

Models are allowed to reason and suggest.

Only the Orchestrator can approve a next step.

Only the Executor can call tools.

No specialist model should directly write files, run shell commands, click the GUI, or modify the system.

The rule is:

```text
User talks to Orchestrator.
Orchestrator talks to agents.
Agents suggest.
Supervisor decides.
Executor acts.
UI displays result and trace.
```

## 4. Model Roles

Use placeholders only. The actual model names should be configured later.

### 4.1 Supervisor Model

Placeholder:

```text
<SUPERVISOR_MODEL>
```

Responsibilities:

* Understand the user request.
* Create an execution plan.
* Decide which specialist agents are needed.
* Decide whether tool use is needed.
* Approve or reject specialist outputs.
* Produce the final answer.
* Maintain task state.
* Avoid directly executing tools.


Responsibilities:

* Write quick implementations.
* Produce small patches.
* Fix obvious bugs.
* Generate tests.
* Explain simple code changes.

The fast coder should be used when the system needs speed over deep architecture.

---

### 4.3 Strong Coding Model

Placeholder:

```text

```

Responsibilities:

* Refactor messy code.
* Solve harder coding tasks.

* Produce cleaner long-term solutions.

The strong coder should be used for project structure, deeper reasoning, and quality improvements.

### 4.4 Reasoning Critic Model

Placeholder:

```text
<REASONING_CRITIC_MODEL>
```

Responsibilities:

* Attack weak reasoning.
* Find hidden assumptions.
* Check whether a plan is incomplete.
* Identify edge cases.
* Find logical flaws in code or task plans.
* Challenge the supervisor before final execution.

* No final authority.
* No shell access.

## 6. Model Server Manager

Create a model server manager that can start, stop, restart, and check local model servers.

The system must support two operating modes.

### 6.1 Always-On Mode

Use when hardware is powerful enough.

```text
All required model servers are started once and kept running.
```

### 6.2 Lazy Sequential Mode

Use on limited hardware.

```text
Only the needed model is started.
The orchestrator calls it.
Then it is stopped or left idle.
The next model is started when needed.
```

Since the system is intended to run sequentially, the initial version should support lazy sequential execution.

Server manager interface:

```python
class ModelServerManager:
    def start_role(self, role: str) -> None:
        pass

    def stop_role(self, role: str) -> None:
        pass

    def restart_role(self, role: str) -> None:
        pass

    def is_role_alive(self, role: str) -> bool:
        pass

    def ensure_role_ready(self, role: str) -> None:
        pass
```

## 9. Orchestrator Responsibilities

The Orchestrator is the main backend service.

It should:

1. Receive the user request.
2. Create or resume a session.
3. Send the request to the supervisor.
4. Ask the supervisor for a structured plan.
5. Route subtasks to specialist agents.
6. Collect specialist outputs.
7. Ask critics to review when needed.
8. Ask supervisor to decide next step.
9. Execute approved tool calls through Executor.
10. Return observations to supervisor.
11. Continue until task is complete.
12. Return final answer to the user.
13. Save trace, messages, tool calls, and results.

The Orchestrator should not be a simple chatbot wrapper. It should be a task controller.

## 11. Orchestration Modes

Implement several orchestration modes.

### 11.1 Simple Mode

Use only supervisor.

```text
User → Supervisor → Final Answer
```

Use for simple questions.

---

### 11.2 Coding Mode

Use supervisor, two coders, critic, and final supervisor.

```text
User
 ↓
Supervisor creates plan
 ↓
Fast coder creates implementation
 ↓
Strong coder reviews/improves
 ↓
Reasoning critic finds flaws
 ↓
Supervisor synthesizes final answer
```

Use for coding tasks.

---

### 11.3 Deep Review Mode

Use all non-vision agents.

```text
User
 ↓
Supervisor plan
 ↓
Fast coder
 ↓
Strong coder
 ↓
Reasoning critic
 ↓
Contrarian critic
 ↓
Supervisor final decision
```

Use for complex tasks or when user asks for high reliability.

---

### 11.4 Tool Execution Mode

Use supervisor and executor.

```text
User
 ↓
Supervisor plan
 ↓
Specialist agents if needed
 ↓
Supervisor proposes tool call
 ↓
Safety validator approves/rejects
 ↓
Executor runs tool
 ↓
Observation returns
 ↓
Supervisor decides next step
```

Use for file writing, shell commands, and later GUI control.

---

### 11.5 GUI Vision Mode

Use screenshot tool, vision agent, supervisor, and GUI executor.

```text
User task
 ↓
Screenshot
 ↓
Vision model describes screen
 ↓
Supervisor decides one next action
 ↓
Safety validator checks action
 ↓
Executor performs one action
 ↓
Screenshot again
 ↓
Repeat until complete
```

Important rule:

```text
Only one GUI action per loop.
Always observe after acting.
```

## 14. Tool System

The tool system must be deterministic and controlled by the Executor.

Every tool must have:

```text
name
description
input_schema
permission_level
validator
execution_function
result_schema
```

Base tool interface:

```python
from abc import ABC, abstractmethod
from pydantic import BaseModel

class ToolResult(BaseModel):
    success: bool
    output: str | None = None
    error: str | None = None
    data: dict | None = None

class BaseTool(ABC):
    name: str
    description: str
    permission_level: int

    @abstractmethod
    def validate(self, arguments: dict) -> None:
        pass

    @abstractmethod
    def execute(self, arguments: dict) -> ToolResult:
        pass
```

## 15. Initial Tools

### 15.1 Filesystem Tools

```text
list_dir(path)
read_file(path)
write_file(path, content)
append_file(path, content)
delete_file(path)
make_dir(path)

```

Restrictions:

* Initially only allow paths inside `/workspace`.
* Block absolute paths unless explicitly approved.
* Block writing outside the project workspace.
* Block deleting directories in MVP.

---

### 15.2 Shell Tools

Implement after filesystem tools:

```text
run_command(command, cwd, timeout)
```

Restrictions:

* Default working directory must be `/workspace`.
* Block destructive commands by default.
* Block commands touching system directories.
* Require confirmation for package installs.
* Require confirmation for network commands.
* Timeout every command.
* Capture stdout, stderr, and exit code.

## 16. Safety System

Safety should be implemented outside models.

Create `config/safety.yaml`:

```yaml
safety:
  default_mode: "safe"

  permissions:
    filesystem_read: true
    filesystem_write_workspace: true
    filesystem_write_outside_workspace: false
    shell_read_only: true
    shell_write_workspace: true
    shell_network: false
    shell_package_install: "confirm"
    gui_control: false
    browser_control: false
    destructive_actions: "confirm"

  blocked_paths:
    - "C:/Windows"
    - "C:/Program Files"
    - "C:/Program Files (x86)"
    - "C:/Users/<USER>/AppData"
    - "/bin"
    - "/etc"
    - "/usr"
    - "/System"
    - "/Library"

  blocked_command_patterns:
    - "rm -rf"
    - "del /s"
    - "format"
    - "shutdown"
    - "reg delete"
    - "diskpart"
    - "powershell -enc"
    - "curl * | sh"
    - "wget * | sh"

  confirmation_required:
    - "delete_file"
    - "run_command_with_network"
    - "install_package"
    - "gui_control"
    - "write_outside_workspace"
```

Safety levels:

```text
Level 0: Chat only
Level 1: Read files inside workspace
Level 2: Write files inside workspace
Level 3: Run safe shell commands inside workspace
Level 4: GUI control
Level 5: Destructive actions with explicit confirmation
```

The Executor must check every tool call against this policy.

## 21. Orchestration Algorithms

### 21.1 Auto Mode Algorithm

```python
async def handle_user_message(session_id, message):
    session = get_or_create_session(session_id)

    classification = await supervisor_classify(message)

    if classification.task_type == "simple":
        return await run_simple_mode(session, message)

    if classification.task_type in ["coding", "debugging", "code_review"]:
        return await run_coding_mode(session, message)

    if classification.task_type == "planning":
        return await run_deep_review_mode(session, message)

    if classification.task_type in ["filesystem_task", "shell_task"]:
        return await run_tool_execution_mode(session, message)

    if classification.task_type in ["vision_task", "gui_task"]:
        return await run_gui_vision_mode(session, message)

    return await run_simple_mode(session, message)
```

---

### 21.2 Coding Mode Algorithm

```python
async def run_coding_mode(session, user_message):
    plan = await supervisor.create_plan(user_message)

    fast_output = await coder_fast.run(
        user_message=user_message,
        supervisor_plan=plan
    )

    strong_output = await coder_strong.run(
        user_message=user_message,
        supervisor_plan=plan,
        fast_output=fast_output
    )

    critic_output = await reasoning_critic.run(
        user_message=user_message,
        supervisor_plan=plan,
        fast_output=fast_output,
        strong_output=strong_output
    )

    final_answer = await supervisor.finalize(
        user_message=user_message,
        plan=plan,
        specialist_outputs=[
            fast_output,
            strong_output,
            critic_output
        ]
    )

    save_trace(session)
    return final_answer
```

---

### 21.3 Deep Review Mode Algorithm

```python
async def run_deep_review_mode(session, user_message):
    plan = await supervisor.create_plan(user_message)

    fast_output = await coder_fast.run(user_message, plan)
    strong_output = await coder_strong.run(user_message, plan, fast_output)
    critic_output = await reasoning_critic.run(user_message, plan, strong_output)
    contrarian_output = await contrarian_critic.run(
        user_message,
        plan,
        fast_output,
        strong_output,
        critic_output
    )

    final_answer = await supervisor.finalize(
        user_message=user_message,
        plan=plan,
        specialist_outputs=[
            fast_output,
            strong_output,
            critic_output,
            contrarian_output
        ]
    )

    return final_answer
```

---

### 21.4 Tool Execution Algorithm

```python
async def run_tool_execution_mode(session, user_message):
    plan = await supervisor.create_plan(user_message)

    while not task_complete:
        next_step = await supervisor.decide_next_step(
            user_message=user_message,
            current_plan=plan,
            previous_observations=session.observations
        )

        if next_step.type == "final":
            return next_step.answer

        if next_step.type == "tool_call":
            validation = safety_validator.validate(next_step.tool_call)

            if validation.requires_confirmation:
                return create_confirmation_request(validation)

            if not validation.allowed:
                observation = {
                    "success": False,
                    "error": validation.reason
                }
            else:
                observation = executor.execute(next_step.tool_call)

            session.observations.append(observation)
            continue
```

---

### 21.5 GUI Vision Algorithm

```python
async def run_gui_vision_mode(session, user_message):
    while not task_complete:
        screenshot = screenshot_tool.execute({})

        vision_observation = await vision_agent.observe(
            image_path=screenshot.path,
            user_goal=user_message
        )

        next_action = await supervisor.decide_gui_action(
            user_goal=user_message,
            vision_observation=vision_observation,
            previous_actions=session.actions
        )

        if next_action.type == "final":
            return next_action.answer

        validation = safety_validator.validate(next_action)

        if validation.requires_confirmation:
            return create_confirmation_request(validation)

        if validation.allowed:
            result = executor.execute(next_action)
        else:
            result = {
                "success": False,
                "error": validation.reason
            }

        session.actions.append(next_action)
        session.observations.append(result)
```

Important:

```text
Never allow multiple GUI actions in a single model response.
Always screenshot after each GUI action.
```

## 24. Error Handling

The system should handle:

```text
model server offline
model timeout
invalid JSON from model
tool validation failure
tool execution failure
user confirmation denied
vision model uncertainty
shell timeout
file permission error
```

For invalid JSON:

1. Try to parse with tolerant parser.
2. Ask the same model to repair JSON.
3. Fall back to plain-text handling.
4. Log the failure.

For model timeout:

1. Retry once.
2. Restart model server if configured.
3. Return a useful error to UI.
4. Save trace.
